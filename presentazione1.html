Contesto

Perché Zero Trust?

Il modello di sicurezza perimetrale tradizionale è obsoleto. Con smart working, IoT e cloud ibrido, il confine della rete non esiste più.

Il Caso di Studio

Protezione di un database ospedaliero (MongoDB) contenente cartelle cliniche e dati sensibili dei pazienti. L'accesso avviene sia dalla rete interna (medici) sia in smart working.

  • Dati sanitari sensibili (GDPR, NIS2)
  • Accesso multi-zona (ospedale + remoto)
  • Necessità di autenticazione continua

Il Paradigma Zero Trust

"Never Trust, Always Verify"

  • Nessun utente è intrinsecamente fidato
  • Ogni richiesta viene autenticata e autorizzata
  • Accesso basato sul contesto e rischio in real-time
  • Micro-segmentazione netta

Requisiti del Progetto

Envoy PEP OPA PDP Splunk SIEM NFTables Snort NIDS MongoDB Machine Learning

Identità

Le 6 Dimensioni Zero Trust

Ogni singola richiesta viene profilata analizzando 6 dimensioni ortogonali prima di concedere l'accesso.

👤 User (u)

Identità utente dal campo CN del certificato mTLS. Es: alice, bob

🖥️ Device (d)

Stato hardware. Verifica dell'OID TPM 1.3.6.1.4.1.9999.1 nel certificato X.509.

💻 Software (s)

Fingerprint JA3 calcolato dall'hash del TLS ClientHello.

🌐 Network (n)

IP sorgente. Distingue rete interna da rete esterna.

⚡ Action (a)

Operazione: metodo HTTP o operazione MongoDB (find, insert).

📁 Resource (r)

Destinazione: endpoint API o collection MongoDB (es. patients).

La Tupla ZTA inviata ad OPA

T = ( u, d, s, n, a, r )

Policy Enforcement Point

Envoy Proxy — Listener

L'unica porta blindata tra i client e le risorse. Ogni pacchetto transita attraverso Envoy, che valida, ispeziona e autorizza.

MongoDB — Porta 27017

  • TLS Inspector con fingerprinting JA3
  • mTLS obbligatorio (require_client_cert: true)
  • Filtro mongo_proxy con BSON parser
  • Filtro ext_authz L4 gRPC verso OPA
  • Certificato inviato ad OPA per analisi (TPM)

HTTP — Porta 8443

  • Header Mutation: inietta hash JA3 calcolato
  • Filtro Lua L7: blocca /api/patients/sensitive
  • Filtro Lua L7: blocca metodo DELETE
  • Filtro ext_authz L7 gRPC verso OPA
  • Access log JSON strutturati (6D ZTA)

Policy Decision Point

Open Policy Agent — Regole

Il cervello decisionale. Riceve la tupla 6D da Envoy, valuta il rischio con Splunk ML e applica DPI.

Funzioni Base

  • Deny-by-default: default allow = false
  • Estrazione identità da certificato mTLS
  • Parsing X.509 per verifica OID TPM
  • Deep Packet Inspection (L7 DPI) MongoDB
  • Integrazione asincrona Splunk MLTK
rules.rego 
# Verifica presenza chip TPM hardware
is_tpm := true if {
  cert_pem := urlquery.decode(cert_raw)
  certs := crypto.x509.parse_certificates(cert_pem)
  some ext in certs[0].Extensions
  ext.Id == [1,3,6,1,4,1,9999,1]
}

Firewall L3/L4

NFTables — Firewall

Il primo livello di difesa. Filtra il traffico L3/L4, gestisce il NAT verso Envoy e blocca bypass diretti.

Funzionalità

  • DNAT: redirige porta 8443 su Envoy
  • Masquerade (SNAT)
  • Log + DROP bypass verso API
  • Set denylist per ban dinamici
  • Integrazione log Fluent Bit
start.sh 
# DNAT: devia porta 8443 su Envoy
nft add rule ip nat prerouting \
  tcp dport 8443 counter dnat to $ENVOY:8443

# TRAPPOLA: Drop per bypass API
nft add rule ip nat prerouting \
  tcp dport 8000 drop

Network Intrusion

Snort — NIDS

Monitoraggio passivo per identificare scansioni di rete o tentativi di movimento laterale.

Caratteristiche

  • Sniffing real-time sui pacchetti
  • Condivisione stack rete Envoy
  • Rilevamento scansioni (Nmap)
  • Output JSON unificato
local.rules 
# Rileva scansione Nmap (SYN)
alert tcp any any -> any any (
  msg:"ZTA ALERT: Scansione Rete";
  flags:S;
  threshold: track by_src, count 20;
  sid:1000001;
)

# Bypass MongoDB
alert tcp any any -> any 27017 (
  msg:"ZTA ALERT: Accesso Diretto DB!";
  sid:1000002;
)
